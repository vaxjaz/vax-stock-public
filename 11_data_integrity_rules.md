# 11 数据严谨性硬规矩

> **建立**: 2026-06-19 · **立规人**: Vax · **适用对象**: Claude(每次分析/查代码/改脚本)
> **本文件是 P0 级硬约束**,优先级等同 00_README 的 SSOT 原则。违反即视为重大失误。
> **触发背景**: 2026-06-19 排查主力净流入 + 融资比两个 bug 期间,Claude 多次把推测当结论
> (误判"砍10000倍"实为×1000、误断"回测报告口径不同"、用无依据阈值"<2000亿=残缺"),

## 一、核心原则:数据与计算逻辑必须 100% 可追溯,禁止猜测

任何关于**指标数值、脚本计算逻辑、字段单位、信号判定**的结论,必须满足:

1. **可追溯到源头**:源码某一行 / 官方文档明确定义 / 用户实测数据。三者之一,缺一不可下定论。
2. **不得用"大概/可能/应该/通常"包装结论**。这些词出现 = 正在猜 = 必须停下,要么去查证,要么明确标注"待验证"。
3. **结论必须区分两态**:
    - ✅ **已证实**:有源码/文档/数据支撑,可以说死
    - ⚠️ **待验证**:手上没有足够依据,必须明示"这需要 XX 验证",并给出验证方法,绝不当成定论

## 二、四条具体禁令

| 禁令 | 说明 | 反面教训(2026-06-19) |
|---|---|---|
| **禁止凭记忆/印象判断代码行为** | 函数定义不在手上 → 必须索取源码,不得猜内容 | 曾断言"回测与报告主力定义不同/符号相反",未读源码,后收回 |
| **禁止用无依据的阈值/标准** | 任何判读阈值必须有出处;自己拍的阈值必须标"假设,待验证" | 曾用"<2000亿=残缺"判断,纯属拍脑袋,差点误判2024年8月真实地量数据 |
| **禁止把单位/量级换算想当然** | 单位必须查官方文档字段说明确认 | 主力净流入曾误判"砍10000倍",实为×1000应为×10000(10倍) |
| **禁止聚合数据不校验来源完整性** | 多来源(多交易所/多市场)聚合前必须校验是否齐全 | 融资比 groupby.sum 未校验交易所,残缺日分子腰斩致信号翻转 |

## 三、数据"未就绪" vs "缺失" 的处理准则

(源自融资比 bug 的教训,通用于所有时变指标)

- **未就绪**(数据迟发布,如盘中拉取时某交易所未公布):**往前推,用最近一个完整交易日的真值,并在报告标注数据日期 + 未就绪状态**。慢一天的真数据 > 当天的坏数据。
- **缺失**(数据源本身没有):同样往前推 + 标注;若连续多日取不到,**报错提醒人工介入**,不得静默用旧值假装正常。
- **绝对禁止**:用已有部分数据推算/估算缺失部分(如"用SSE按占比反推总额")。推算 = 编造假数据。

## 四、Claude 自检清单(每次给数据相关结论前过一遍)

1. 这个结论的依据是源码第几行 / 哪份文档 / 哪条实测数据?说不出 → 不能下结论
2. 我用了"可能/大概/应该"吗?用了 → 要么查证,要么标"待验证"
3. 我引用的单位/量级,查过官方字段定义吗?没查 → 先查
4. 这是"已证实"还是"待验证"?必须让用户看得出是哪一种
5. 我给的判读阈值,有出处吗?自己定的 → 标注"假设,需数据校验"
6. 涉及多来源聚合 → 校验过完整性吗?
